Feature release adding an opt-in speech queue for overlapping voice alerts,
plus CI test fixes, on top of `v2026.07.2`.

## Highlights

### Opt-in speech queue for overlapping voices

- New `cn voice queue on|off|status` command serializes concurrent
  utterances through an on-disk mutex, so parallel agents finishing at the
  same time speak one at a time instead of over each other (macOS only;
  off by default).
- Identical phrases arriving within `CODE_NOTIFY_SPEECH_DEDUP_SECONDS` are
  deduplicated, and phrases waiting longer than
  `CODE_NOTIFY_SPEECH_MAX_WAIT_SECONDS` are dropped; both are logged to
  `notifications.log`.
- Locks held by crashed, hung, or pid-recycled speakers are reclaimed
  safely via conditional mutex-guarded removal.
- Identical concurrent cache-miss synthesis collapses into a single
  ElevenLabs call, with retry backoff after a failed synthesis
  (`CODE_NOTIFY_TTS_FAIL_BACKOFF_SECONDS`).
- Synthesis is prefetched before taking the speech lock, so the lock is
  held for playback only.
- The Windows installer reports the queue as unsupported.

## Test and CI fixes

- Fixed the Windows wording test collapsing its heredoc-extraction regexes
  under PowerShell string expansion, which made it validate the wrong
  script and fail on CI runners with pwsh installed.
- Scoped spoken-output assertions in the wording-style test to macOS;
  the Linux notifier branch sends banner and sound only, so waiting for
  spoken output always timed out on Linux CI.
